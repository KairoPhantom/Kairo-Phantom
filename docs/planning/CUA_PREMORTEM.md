# Kairo Phantom — CUA 1000x Upgrade Premortem

This document provides a pre-mortem analysis of the Computer Use Agent (CUA) 1000x Upgrade. It outlines the 15 major risks (R-01 through R-15) that could impact reliability, safety, usability, or performance, along with their respective mitigation plans.

---

### R-01: Hotkey Conflict
* **Risk:** The CUA trigger hotkey (`Alt+Ctrl+M`) conflicts with global OS hotkeys or popular third-party applications, preventing the user from activating ghost-writing.
* **Mitigation:**
  - Implemented low-level hook `WH_KEYBOARD_LL` on Windows which intercepts keys before application-level event handlers.
  - Provided custom hotkey configuration in `config.toml` to allow users to override the defaults.
  - Active hotkey is validated at startup to ensure it doesn't collide with reserved system shortcuts.

### R-02: DPI Scaling and Coordinate Drift
* **Risk:** High-DPI screens or multi-monitor configurations cause mouse click coordinates generated by the VLM to drift, clicking in the wrong locations.
* **Mitigation:**
  - Standardized all screenshots to logical coordinates by querying physical monitor scale factors via Tauri/Windows APIs.
  - Implemented the DPI scaling test `test_dpi_scaling_applied_to_mouse_coordinates` to ensure scale calculations remain correct.
  - Coordinate offsets are dynamically adjusted relative to the bounding box of the active window.

### R-03: VLM Downtime / Connection Failure
* **Risk:** The local Ollama server fails to start, or crashes mid-session, making the Tier 3 visual grounding fallback unavailable.
* **Mitigation:**
  - Implemented a robust "keyboard-only" fallback mode. If VLM is unreachable, the executor uses preset shortcut templates (`cua_shortcuts.toml`) to complete standard actions.
  - Sidecar client checks Ollama server health via periodic ping requests and reports status to the tray.
  - Automatic sidecar auto-restart loop is triggered if the connection drops.

### R-04: Slow Inference / Latency
* **Risk:** Local VLM model execution takes too long on lower-end systems, causing lag and ruining the "instant AI" UX.
* **Mitigation:**
  - Implemented a hardware-adaptive model picker: ≥6GB VRAM loads the 7B model, while <6GB loads the lightweight 3B model.
  - CPU-only systems automatically enforce the 3B GGUF model and use optimized quantizations to minimize latency.
  - LiteLLM warms up standard models at startup in a separate background preloader thread.

### R-05: Rollback Failure / Incomplete Undo
* **Risk:** An action fails halfway, but the rollback mechanism fails to revert typing or file changes, leaving document states corrupted.
* **Mitigation:**
  - Maintained a rollback transaction log tracking all keyboard modifications and file actions.
  - Keyboard modifications are reverted using `Ctrl+Z` undo emulation, and files are restored from temporary backup snapshots.
  - File backups are verified using SHA-256 hashes before restoring to prevent writing outdated backups.

### R-06: PII / Password Leakage in Logs
* **Risk:** The workflow recorder captures sensitive information, such as passwords or personal data, and writes it to plain-text TOML logs.
* **Mitigation:**
  - Implemented aggressive filtering in `app_watcher.rs` to detect and ignore password fields using UIA elements' `is_password` flags and class-name checks.
  - Cleared intermediate keystroke buffers immediately after serialization.
  - Enforced local-only log directories with strict OS file permissions.

### R-07: Registry Deserialization Failures
* **Risk:** Custom community shortcut registries downloaded by `kairo shortcuts update` are malformed, crashing the parser on startup.
* **Mitigation:**
  - Implemented schema-based validation on download before overwriting the active `cua_shortcuts.toml`.
  - Kept a hardcoded fallback registry of standard shortcuts in the binary to ensure execution is never blocked.
  - Logged detailed parse errors and automatically reverted to the last known-good configuration.

### R-08: Cross-App Security Boundary Violations
* **Risk:** CUA commands execute actions in unauthorized, hidden, or forbidden windows (e.g. system terminal, password managers).
* **Mitigation:**
  - Implemented window blacklist filters in the `CuaGate` checker to block execution on disallowed window titles (e.g., "Registry Editor", "Task Manager").
  - Blocked interaction with password input fields at the UIA level during step verification.
  - Visual GRP approval window forces human-in-the-loop review for any app context changes.

### R-09: Infinite Input Loops
* **Risk:** CUA keyboard emulation triggers a sequence of keys that recursively triggers the Kairo hotkey, causing a runaway loop.
* **Mitigation:**
  - Implemented event debouncing and ignored inputs sent programmatically by the process itself.
  - Hook listener temporarily suspends hotkey detection during active CUA emulation sequences.
  - Hardcoded safety limit on sequence length and execution time.

### R-10: VLM False Positives in Verification
* **Risk:** The semantic verifier incorrectly reports a successful action when the screen state has actually not changed.
* **Mitigation:**
  - Combined VLM verification with low-level UIA property checks to cross-verify.
  - Configured prompt templates to return a confidence score alongside reasoning.
  - Actions with a verification confidence score below 80% default to a warning and request user approval.

### R-11: UIA Element Tree Disruption
* **Risk:** The active application's UI elements are updated or re-rendered mid-sequence, causing UIA element cache invalidation and step failure.
* **Mitigation:**
  - Implemented dynamic re-queries for elements if the cached handle becomes stale.
  - Used stable window/control-type query selectors rather than raw memory pointers.
  - Gracefully fallback to visual coordinate clicking (VLM grounding) if UIA searches time out.

### R-12: Keyboard Layout Mismatches
* **Risk:** Simulated keystrokes map to incorrect characters on international or non-US keyboard layouts.
* **Mitigation:**
  - Enigo input utilizes unicode-character simulation rather than raw virtual keycodes where supported.
  - Added keyboard layout checking at startup to warn users of potential mismatch issues.
  - Set fallback typing delay to allow OS key state buffers to settle.

### R-13: Race Conditions in Async Execution
* **Risk:** UI interactions trigger too fast, sending keyboard inputs before the target window has completely gained focus.
* **Mitigation:**
  - Enforced a configurable `typing_delay_ms` delay between individual keys.
  - CuaExecutor explicitly waits for the target window to be in the foreground before typing.
  - Verified window focus state via OS API calls before each action.

### R-14: OS Compatibility Gaps
* **Risk:** Features like WH_KEYBOARD_LL hooks fail to compile or function on macOS/Linux targets.
* **Mitigation:**
  - Enforced platform-specific conditional compilation (`#[cfg(windows)]`, `#[cfg(target_os = "macos")]`, etc.).
  - Used cross-platform libraries like `rdev` for non-Windows targets.
  - Excluded Windows-only tray notification APIs on other platforms.

### R-15: Driver / Hook Blocking by Security Software
* **Risk:** Antivirus/EDR tools flag low-level keyboard hooks or mouse emulation drivers as suspicious behavior, blocking Kairo.
* **Mitigation:**
  - Signed all binaries with a local developer certificate during installation.
  - Modeled keyboard/mouse actions to mimic human timing and patterns (natural curves and delay variance).
  - Provided clear administrative installation guidelines and whitelist profiles.

### R-16: Suffix Automaton State Explosion
* **Risk:** Suffix Automaton constructed from extremely large corpora consumes too much memory, leading to client-side memory exhaustion.
* **Mitigation:**
  - Standardized on word-level tokens instead of character-level transitions to keep states bounded.
  - Capped maximum length of texts registered into the automaton and focused on high-risk phrases.
  - Implemented lightweight Bloom filters as a first-pass membership test before traversing automaton paths.

### R-17: Differential Privacy Epsilon Exhaustion
* **Risk:** Frequent FedAvg updates deplete the local privacy budget (epsilon, delta) too quickly, locking out memory sync features.
* **Mitigation:**
  - Rate-limited federated synchronizations to daily/weekly schedules instead of continuous real-time updates.
  - Logged all epsilon allocations to the SHA-256 tamper-evident audit chain to track usage.
  - Applied Gaussian Mechanism with L2 clipping of centroids to maximize privacy amplification.

### R-18: Data Provenance Check Bypass
* **Risk:** Model adapters are trained on non-compliant, copyrighted datasets because of bypasses or missing checks in the SFT training loop.
* **Mitigation:**
  - Enforced strict `data_provenance` validation checks directly in the SFT pipeline startup code.
  - Code raises a blocking `ValueError` and refuses to train if any dataset row is missing the proper `synthetic-opt-in` or `user-opt-in` tags.
  - Verified and shuffled datasets automatically inside containerized CI boundaries.

### R-19: World Model UIA Tree Inconsistency
* **Risk:** Rapid UI changes in active applications cause the World Model tree state cache to drift from reality, resulting in incorrect clicks or actions.
* **Mitigation:**
  - Configured standard UIA polling at 250ms with element subtree diffing.
  - Combined cache lookups with immediate verification checks (`verify_action`) to confirm side-effects.
  - Action failures automatically trigger complete invalidation of the tree cache to force a fresh perception loop.
