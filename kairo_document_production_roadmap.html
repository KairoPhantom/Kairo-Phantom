Kairo Phantom
Document-Complete Roadmap

Eight document categories. Every format fully functional — read, understand, generate, inject. Not done until a user can press Alt+M in any app, on any document, and get output they can rely on without editing. No clipboard hacks. No formatting loss. No hallucinations about structure.

00 Why everything is currently broken

Before you write a single line, understand the three root causes behind every bad output you've seen in testing. Every fix in this roadmap addresses one of these.

✗
Root cause 1 — Context is flat, structure is invisible. UIAutomation gives plain text with no heading hierarchy, no table boundaries, no list markers, no paragraph styles. The LLM sees "Introduction Background Results" as prose, not as three separate Heading 2s. It has no idea what structural role any text plays. Every output is a structural guess.
✗
Root cause 2 — Output is freeform, injection is raw. The LLM produces natural language prose. The injector pastes it via clipboard. Word receives plain text with no paragraph styles, no heading levels, no run formatting. Even if the AI writes perfect content, it arrives as a wall of unstyled normal text.
✗
Root cause 3 — No format-specific write-back. Each document type (DOCX, XLSX, PPTX, PDF) has a completely different write model. There is zero per-format write logic in the current injector — everything goes through the same clipboard/SendInput path. You cannot achieve correct formatting in Excel or PowerPoint using clipboard paste from a text stream.
✗
Root cause 4 — 7B model with zero structural constraint. Asking a 7B model to simultaneously understand context, respect style, match tone, and produce correctly-formatted output is asking it to do four hard things at once with no guardrails. It will fail inconsistently. The fix is structured output schemas that constrain what the model can produce per document type — not better prompts.
01 The production architecture

Three layers. Every document type goes through all three. Nothing bypasses the pipeline. This replaces the current UIA → clipboard path for all saved documents.

The Python sidecar model (IPC via stdin/stdout or Unix socket)
Your Rust daemon cannot natively use Docling, python-docx, openpyxl, or python-pptx — they're Python libraries. The right architecture is a persistent Python sidecar process that Rust spawns on startup and communicates with over a local socket or stdin/stdout pipe. The sidecar owns all document I/O. Rust owns the hotkey, UIA, injection, and orchestration. This keeps your Rust daemon lean and fast while giving you access to the entire Python document ecosystem without process-per-request overhead.

Layer 1 · Read
Docling (IBM) — unified parser
Converts PDF, DOCX, PPTX, XLSX, HTML, images → structured DoclingDocument JSON. Preserves headings, tables, reading order, lists. Replaces flat UIA extraction for all file-based documents. 37k GitHub stars, MIT licensed, runs 100% locally.
Layer 2 · Generate
Structured schema + LiteLLM gateway
Each document type has a typed JSON schema (DocOperation, SlideOperation, CellOperation). LLM is forced to output valid operations, never freeform prose. LiteLLM routes: local Ollama primary, cloud fallback. XGrammar constrains output at token level for local models.
Layer 3 · Write
Format-specific writers
python-docx for DOCX (paragraph styles, runs, tables). openpyxl for XLSX (cells, formulas, conditional formatting). python-pptx for PPTX (text frames, layouts, charts). WeasyPrint for PDF generation. Each writer applies operations with correct formatting — never raw text injection.
Fallback · UIA + Clipboard
Existing path — for unsaved / browser / terminal contexts
Keep the existing UIA reader and clipboard injector exactly as-is for contexts where there is no file on disk: web browsers, terminal applications, chat clients, Notepad, and any app where file path is unavailable. The new sidecar path only activates when a document file can be resolved. Two paths, both maintained.
Cross-cutting
MinerU 3.1 — complex document fallback
For PDFs with scanned content, multi-column layouts, or complex tables where Docling struggles, MinerU 3.1 (1.2B ONNX model) handles high-precision extraction. State-of-the-art on OmniDocBench, beats GPT-4o on table recognition, 109 languages.

06 What to explicitly not build yet

Every item below is a real temptation. Don't touch them until all 8 formats pass their acceptance tests.

✗
Stable Diffusion integration — image generation adds zero value until text generation is reliable. Users won't notice or use image generation in documents if the text quality is still poor. This is a Phase 4+ feature at the earliest.
✗
Web scraping / live data fetching — the MCP/Context7 integration can wait. Users need reliable offline document handling first. Every new integration adds surface area for failures before you have solid baselines.
✗
macOS port — only after the Windows document stack is production-ready and passing E2E tests. Porting before you have a solid base means you'll be debugging two platforms simultaneously.
✗
Multi-model agent routing upgrades — the Swarm Orchestrator improvement can wait. Fix the read/write pipeline first. A better agent routing to the same bad context extraction produces the same bad output.
